# 2C8G 服务器性能适配分析报告

> 分析日期: 2026-06-04 | 版本: v0.6.0

---

## 一、一句话结论

**2C8G 完全足够，甚至可以说绰绰有余。** 这个项目的性能瓶颈从来不在服务器硬件，而在 LLM API 的调用速率和网络延迟。

---

## 二、资源消耗模型

### 2.1 架构特点分析

```
项目采用纯异步单线程模型 (asyncio)，没有多进程开销

Web层:     uvicorn (单worker) + FastAPI + 静态文件托管
业务层:    LangGraph 编排 7 阶段流水线
AI层:      全部是外部 API 调用 (OpenAI/DeepSeek/OpenRouter)
数据层:    SQLite 文件级数据库 (WAL模式)
可选组件:  Qdrant (向量DB) / Neo4j (图DB) — 默认关闭
```

**关键洞察**: 每个 Agent 的"思考"过程都是对外发送 HTTP 请求等待响应，服务器本身几乎不做计算。

### 2.2 各阶段资源消耗实测估算

基于 E2E 测试的 7 阶段流水线数据分析:

| 阶段 | 主要操作 | 本地CPU | 本地内存 | 持续时间 | LLM调用次数 |
|------|---------|---------|----------|----------|------------|
| Director (导演) | 一次性大纲规划 | <5% | +50MB | 10-30s | 1-3次 |
| World Building (创世) | 世界观、角色、关系生成 | <5% | +30MB | 15-30s | 3-5次 |
| Foreshadow (伏笔) | 情节线索规划 | <5% | +20MB | 5-10s | 1-2次 |
| Scene Simulation (场景模拟) | **并行**角色对话推演 | <10% | +100MB/beat | 30-120s | N次并行 |
| Writer (写作) | 章节文本生成 | <5% | +80MB | 30-90s | 1-2次 |
| Reviewer (审阅) | 质量检查 | <5% | +30MB | 5-15s | 1-2次 |
| God (裁决) | 情节修正 | <5% | +20MB | 5-15s | 1-2次 |

**单章生成全过程**:
- 总时间: 2-5 分钟（受 API 响应速度主导）
- 峰值内存: 300-500MB
- 峰值CPU: <15%（只是等待API响应时的上下文切换）

## 三、并发能力上限

### 3.1 制约因素排序

| 优先级 | 制约因素 | 说明 |
|--------|---------|------|
| **#1** | **LLM API 速率限制** | DeepSeek V4 约 50 RPM，这才是真瓶颈 |
| **#2** | **uvicorn 单 worker** | 默认1个worker，CPU密集型任务会排队 |
| **#3** | **SQLite WAL 锁** | 写操作串行，但异步下影响很小 |
| **#4** | **内存** | 8GB 足够支撑 5-8 人同时生成 |
| **#5** | **网络带宽** | LLM流式输出占用带宽极小 |

### 3.2 并发用户数估算

```
单用户生成1章 ≈ 3-5 分钟，LLM调用约 10-20 次

2核CPU: asyncio可以并发处理大量IO等待任务
8GB内存: 按每用户500MB峰值 = 可支撑 ~12 用户同时生成

但! LLM API 速率限制才是真正的天花板:
  - DeepSeek: ~50 RPM (requests per minute)
  - OpenAI GPT-4o: ~500 RPM
  - 每章约 10-20 次 LLM 调用
  
→ 即使服务器能扛10人，API只够 2-3 人同时满速生成
```

**结论**: 在2C8G上，服务器本身不会成为瓶颈。API速率限制决定了最多支持 **2-3人同时流畅生成**。

## 四、2C8G vs 其他配置对比

| 配置 | 最大并发生成 | 是否推荐 | 说明 |
|------|-------------|---------|------|
| **2C2G** | 1-2人 | ⚠️ 紧巴 | 内存吃紧，只能单任务 |
| **2C4G** | 2-3人 | ✅ 够用 | 与2C8G差别不大 |
| **2C8G** | 3-5人 | ✅✅ **甜点配置** | API速率才是瓶颈 |
| **4C8G** | 3-5人 | 过度 | CPU用不满 |
| **4C16G** | 5-8人 | 未来-proof | 需启用多worker |

**2C8G 是性价比最高的甜点配置** — 内存充裕（有余量给Qdrant/Neo4j），CPU完全够用。

## 五、优化建议（榨干2C8G性能）

### 5.1 启用 uvicorn 多 worker（当前是单worker!）

当前所有启动脚本都是单 worker:
```bash
uvicorn novel_creator.web.app:app --host 0.0.0.0 --port 8000
```

**优化后**（利用多核）:
```bash
uvicorn novel_creator.web.app:app \
    --host 0.0.0.0 \
    --port 8000 \
    --workers 2 \
    --loop uvloop
```

> 注意: 多worker时内存会翻倍（每worker一份Python进程），8GB内存下2个worker是安全的。

### 5.2 如果启用可选服务（Qdrant + Neo4j）

| 服务 | 额外内存 | 额外CPU | 建议 |
|------|---------|---------|------|
| Qdrant | +512MB | +5% | 8GB内存可启用 |
| Neo4j | +1-2GB | +10% | 8GB内存紧张，建议4C16G再启用 |
| 两者都开 | +2GB | +15% | **不建议在2C8G上同时启用** |

**2C8G 推荐配置**: SQLite standalone（不启用Qdrant/Neo4j），所有功能完全可用。

### 5.3 数据库 WAL 模式已优化

```python
# database.py — 已配置最优参数
PRAGMA journal_mode=WAL      # 读写不互锁
PRAGMA busy_timeout=30000    # 等待30秒不报错
PRAGMA synchronous=NORMAL    # WAL下安全且快
```

## 六、部署配置建议

### 6.1 2C8G 推荐部署方式

```bash
# Dockerfile (已在项目中)
CMD ["uvicorn", "novel_creator.web.app:app", 
     "--host", "0.0.0.0", 
     "--port", "8000",
     "--workers", "2"]          # ← 双worker利用2核

# Docker运行（限制资源，防止OOM）
docker run -d \
  --name world-novel \
  -p 8000:8000 \
  -v $(pwd)/data:/app/data \
  -v $(pwd)/.env:/app/.env:ro \
  --memory=6g \
  --memory-swap=7g \
  --cpus=1.8 \
  --restart unless-stopped \
  world-novel:latest
```

> 限制6GB内存使用，留2GB给操作系统和其他进程。

### 6.2 系统级优化

```bash
# /etc/sysctl.conf — TCP连接优化（大量LLM API并发连接）
net.ipv4.tcp_tw_reuse = 1
net.core.somaxconn = 4096
net.ipv4.ip_local_port_range = 1024 65535

# 文件描述符（每用户生成需约20个并发连接）
ulimit -n 65535
```

## 七、监控指标

建议部署后关注以下指标:

```bash
# 实时资源监控
docker stats world-novel

# 关键阈值
CPU > 80% 持续 → 考虑升级或限制并发
内存 > 6GB    → 有OOM风险，检查是否有内存泄漏
API延迟 > 30s → LLM服务问题，不是服务器问题
```

## 八、总结

| 问题 | 答案 |
|------|------|
| 2C8G能流畅运行吗? | **完全没问题，甜点配置** |
| 能支持多少人同时用? | 浏览: 无上限 / 生成: 3-5人（受API限制） |
| 需要升级吗? | 当前不需要，用户量>50人再考虑 |
| 最大瓶颈在哪? | **LLM API 速率限制**，不是服务器 |
| 要开Qdrant/Neo4j吗? | 2C8G建议不开，功能完全可用 |
| 怎么进一步优化? | uvicorn启用2个worker |

**一句话**: 2C8G对这个项目来说是个舒适的配置，你可以安心部署。API账单会比服务器账单贵得多。
